# gzipt — gzip as a language model

`gzipt` generates text using **gzip as its only model**. No neural network, no
training, no parameters. You prime it with a corpus, and it continues a prompt by
searching for the byte sequences that compress best — because *what compresses
well is what the model predicts.*

```bash
gzipt --corpus data/tinyshakespeare.txt --prompt $'MENENIUS:\n' --length 200 --seed 3
```
```
MENENIUS:
'Though all at once canq

MARCIUS:
Pray now, nocamest thou to a morsel .

LARTIUS:
Hence, and
I' the end admire, where G
again; and after it ag .

LARTIUS:
Hence, and
I' the end ad

LARTIUS:
fame and
```

Real, unedited output: recombined *Coriolanus* fragments and speaker labels, with
the odd rough seam (`canq`) — that's the honest texture (see
[How generation works](#how-generation-works) and the limitations below).

## Compression is prediction

A compressor spends few bytes on data it "expects" and many on data it doesn't,
so the compressed length of a continuation is a score: gzip "expects" a
continuation that echoes text in its window, so that continuation compresses
small. `gzipt` generates by **beam search over byte sequences**, scoring each
candidate by `len(gzip(context + candidate))` and committing the most-compressible
span. The corpus primes gzip's 32 KiB window and is the model's entire knowledge.

**What this really is (worth being honest about):** gzip's compressor predicts the
next bytes by matching the recent context against its window — which is exactly
what an *n-gram* does. So gzipt is, in effect, a variable-order n-gram over the
corpus, with gzip as the (lossy, fuzzy) lookup. The output reads like **recombined
corpus fragments** — locally fluent, globally a collage — not original prose. That
is the honest ceiling of "gzip as a language model," and it's the interesting
result: a generic compressor's hidden language model is an n-gram.

## Install & run

```bash
uv run gzipt --help              # via the installed script
uv run python gzipt.py --help    # or just run the file directly
```

One file, pure standard library (just `zlib`); requires Python ≥ 3.10.

## Usage

```bash
gzipt \
  --corpus FILE \         # primes gzip's window — the model's only knowledge
  --prompt "text" \       # seed to continue
  --length 200 \          # bytes to generate
  --horizon 24 \          # beam depth: bytes looked ahead and committed per span
  --beam-width 32 \       # partial continuations kept each step
  --temperature 0.5 \     # 0 = best span; >0 samples the finalists
  --tail 80 \             # generated bytes kept in scoring context (anti-copy)
  --window 30000 \        # corpus bytes shown to gzip (<= 32768)
  --workers 8             # threads for scoring (zlib releases the GIL)
```

The defaults above are tuned for quality; a 200-byte generation runs in ~7 s.

## How generation works

Each step runs a beam search `horizon` bytes deep and commits that span:

1. **Context.** gzip is shown the corpus window plus only the last `--tail` bytes
   of what it has generated. Hiding older output matters: DEFLATE codes *near*
   matches cheaper than far ones, so if gzip could see its whole history it would
   copy and reinforce it — the verbatim-loop trap.
2. **Beam search over bytes.** Keep `--beam-width` partial sequences; extend each
   by every byte that occurs in the corpus; score all by compressed length; keep
   the best. After `--horizon` bytes you have full candidate spans. A single byte
   barely changes the compressed length (whole-byte resolution), so looking a span
   ahead is what lets the real signal register and avoids the greedy whitespace
   collapse.
3. **Commit.** `--temperature 0` commits the most-compressible span; a positive
   temperature samples the finalists for variety.

### Speed

The bottleneck is compression, reused three ways:

* **encoder-state cloning** — the shared context is compressed once per step, then
  `zlib`'s `compressobj.copy()` snapshots that state per candidate so each only
  pays for its own few bytes (byte-identical to recompressing, ~order of magnitude
  cheaper, and unique to `zlib` among stdlib compressors);
* candidates restricted to bytes that occur in the corpus (~256 → ~80);
* candidate scoring spread across threads (`--workers`).

### Tuning the knobs

- **`--horizon`** — the main quality dial. Too small (≤ 4) → whitespace collapse;
  too large → the fixed-width beam degrades into copying one passage verbatim.
  ~20–28 is the sweet spot.
- **`--beam-width`** — wider keeps more good paths alive, costs linearly more.
- **`--temperature`** — higher = more varied (and helps break verbatim loops),
  lower = more compressible/literal. 0.3–0.7 is a good range.
- **`--tail`** — bytes of self-copy allowed; smaller = less looping, more disjoint.
- **`--window`** — more corpus in view = more to draw from, up to gzip's 32 KiB
  limit; a fuller window noticeably reduces degeneracy, at some speed cost.

Known rough edges: output is a corpus collage, not coherent narrative;
committing a whole span leaves occasional **junk at the seams** (`canq`) where a
match runs out; and it readily falls into **verbatim loops** (repeating a phrase
it just emitted) — raise `--temperature` to fight them.
